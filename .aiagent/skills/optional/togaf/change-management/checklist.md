# Change Management Checklist

Quality gates and verification criteria for TOGAF Phase H activities.

---

## Architecture Monitoring Checklist

### Technical Monitoring
- [ ] Architecture health dashboard reviewed
- [ ] Conformance metrics collected
- [ ] Technology lifecycle status checked
- [ ] Security vulnerability scan results reviewed
- [ ] Technical debt metrics assessed
- [ ] Performance trends analyzed
- [ ] Capacity utilization checked

### Business Alignment Monitoring
- [ ] Business strategy updates reviewed
- [ ] Capability utilization tracked
- [ ] Stakeholder feedback gathered
- [ ] Market/competitor changes noted
- [ ] Regulatory updates checked
- [ ] Strategic initiative alignment verified

### Operational Monitoring
- [ ] SLA performance reviewed
- [ ] Operational cost tracking current
- [ ] Incident patterns analyzed
- [ ] User satisfaction metrics collected
- [ ] Capacity planning updated

### Monitoring Outputs
- [ ] Health report produced
- [ ] Issues documented
- [ ] Opportunities identified
- [ ] Change requests generated (if needed)
- [ ] Escalations raised (if needed)

---

## Change Request Intake Checklist

### Request Completeness
- [ ] Requestor identified
- [ ] Title is clear and descriptive
- [ ] Description is comprehensive
- [ ] Change driver explained
- [ ] Urgency justified
- [ ] Initial scope estimated
- [ ] Affected domains identified

### Initial Triage
- [ ] Change type classified
- [ ] Initial scope determined
- [ ] Request logged in register
- [ ] Tracking ID assigned
- [ ] Acknowledgment sent to requestor
- [ ] Assessment SLA established
- [ ] Assessor assigned

---

## Impact Assessment Checklist

### Architecture Analysis
- [ ] Affected artifacts identified
- [ ] Dependencies traced
- [ ] Principle conformance evaluated
- [ ] Target architecture alignment checked
- [ ] Related initiative conflicts identified

### Domain Impact Assessment
- [ ] Business architecture impact assessed
- [ ] Data architecture impact assessed
- [ ] Application architecture impact assessed
- [ ] Technology architecture impact assessed

### Risk Assessment
- [ ] Implementation risks identified
- [ ] Non-implementation risks considered
- [ ] Reversibility evaluated
- [ ] Complexity assessed
- [ ] Precedent implications considered

### Stakeholder Analysis
- [ ] Affected stakeholders identified
- [ ] Impact severity per stakeholder assessed
- [ ] Support/resistance gauged
- [ ] Communication needs identified

### Resource Estimation
- [ ] Effort estimated
- [ ] Skills requirements identified
- [ ] Resource availability checked
- [ ] Timeline estimated
- [ ] Cost estimated

### Assessment Quality
- [ ] Multiple perspectives included
- [ ] Subject matter experts consulted
- [ ] Assessment documented
- [ ] Recommendation justified
- [ ] Evidence provided

## Disposition Decision Checklist

### Pre-Decision
- [ ] Impact assessment complete
- [ ] Recommendation prepared
- [ ] Appropriate governance body identified
- [ ] Decision materials distributed
- [ ] Decision meeting scheduled

### Decision Process
- [ ] Assessment findings presented
- [ ] Risks discussed
- [ ] Stakeholder concerns addressed
- [ ] Alternative approaches considered
- [ ] Resource availability confirmed

### Decision Documentation
- [ ] Decision clearly stated
- [ ] Rationale documented
- [ ] Conditions specified (if applicable)
- [ ] Next steps defined
- [ ] Owners assigned

### Post-Decision
- [ ] Requestor notified
- [ ] Change register updated
- [ ] Decision communicated to stakeholders
- [ ] Next steps initiated

## Emergency Change Checklist

### Immediate Assessment (< 4 hours)
- [ ] Urgency confirmed
- [ ] Impact assessed
- [ ] Containment identified
- [ ] Authority notified

### Emergency Approval (< 24 hours)
- [ ] Emergency authority convened
- [ ] Findings presented
- [ ] Risks acknowledged
- [ ] Approval obtained
- [ ] Decision documented

### Rapid Implementation
- [ ] Fix implemented
- [ ] Results monitored
- [ ] Status communicated
- [ ] Issues escalated if needed

### Post-Implementation (< 1 week)
- [ ] Full impact assessment completed
- [ ] Architecture artifacts updated
- [ ] Root cause analyzed
- [ ] Process improvements identified
- [ ] Lessons learned documented

## Phase Entry Criteria

Before beginning Phase H activities:

- [ ] Architecture implemented (Phase G complete)
- [ ] Governance framework operational
- [ ] Monitoring capabilities in place
- [ ] Change management process defined
- [ ] Roles and responsibilities assigned
- [ ] Communication channels established

---

## Phase Exit Criteria

Phase H is ongoing, but key checkpoints:

### Per Change Request
- [ ] Change assessed
- [ ] Decision made
- [ ] Change executed (or cycle triggered)
- [ ] Architecture updated
- [ ] Communication complete
- [ ] Request closed

### Periodic (Monthly/Quarterly)
- [ ] Health report produced
- [ ] Issues addressed
- [ ] Metrics reported
- [ ] Governance engaged

## Quality Metrics

### Change Request Metrics
| Metric | Target | Formula |
|--------|--------|---------|
| Processing Time | Within SLA | Submitted → Decided |
| First-Time Approval | >70% | Approved / Decided |
| Assessment Quality | <10% rework | Decisions reversed / Total |

### Monitoring Metrics
| Metric | Target | Formula |
|--------|--------|---------|
| Health Report Timeliness | 100% on schedule | On-time / Scheduled |
| Issue Detection | <5 day lag | Event → Detection |
| Proactive Changes | >50% | Proactive / Total changes |

### Architecture Currency
| Metric | Target | Formula |
|--------|--------|---------|
| Artifact Currency | <30 days stale | Days since last update |
| Documentation Accuracy | >95% | Accurate / Audited |
| Stakeholder Awareness | >80% | Aware / Affected |
